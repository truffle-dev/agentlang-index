# contributing to agentlang-index

This file is for contributors adding tasks to the corpus or improving
the harness. The corpus is the load-bearing artifact, so most of this
document is about how a corpus task is shaped.

## what to contribute

Two surfaces accept contributions:

1. **Corpus tasks.** New task directories under `corpus/<NNN-slug>/`
   following the layout below. Every task ships with five hand-written
   reference implementations plus a verifier that runs them all.
2. **Harness improvements.** `bench/` (TypeScript one-shot runner) and
   `harness/` (Python agent-loop runner). Bug fixes, additional
   provider adapters, scoring refinements, dashboard JSON shape
   adjustments.

What does not accept contributions: the dataset repo
[`truffle-dev/agentlang-index-data`](https://github.com/truffle-dev/agentlang-index-data)
is the authoritative export from this harness, not a contribution
target. The Zero CLI repo
[`truffle-dev/agentlang-spec`](https://github.com/truffle-dev/agentlang-spec)
is in scope for upstream Zero issues, not for agentlang-index task
mechanics.

## adding a new corpus task

A corpus task is one directory under `corpus/` containing a small set
of files in a specific shape. Pick the next free three-digit prefix
and a kebab-case slug. The directory name is the slug.

### required files

```
corpus/<NNN-slug>/
  spec.json              task metadata, prompt, acceptance criteria
  prompt.md              prompt template with {language_scaffold} marker
  ref.zero               Zero reference implementation
  ref.ts                 TypeScript reference implementation
  ref.rs                 Rust reference implementation
  ref.go                 Go reference implementation
  ref.py                 Python reference implementation
  verify.sh              runs every reference and asserts identical output
  notes.md               design rationale, calibration, gotchas
  tests/
    public/case-NNN.json public test cases (shown to the model)
    hidden/case-NNN.json hidden test cases (used for grading)
```

HTTP-flavor tasks may add `fixture_server.py` plus a per-language
package manifest (`Cargo.toml`, `go.mod`, etc.) when the language's
HTTP client needs a build step. Look at `corpus/012-http-status-code/`
for the full HTTP-shape layout.

### spec.json

```json
{
  "slug": "NNN-slug",
  "title": "Human-readable task title",
  "difficulty": "easy | medium | hard",
  "tags": ["category", "...", "..."],
  "spec_version": 1,
  "prompt": "Full problem statement as a single string. This is what the model sees.",
  "input_schema": {"stdin": "..."},
  "output_schema": {
    "stdout": "...",
    "stderr": "",
    "exit_code": 0
  },
  "acceptance": {
    "stdout_exact": "...",
    "stderr_exact": "",
    "exit_code": 0,
    "wall_time_max_ms": 5000
  },
  "token_budget": 500,
  "languages": ["zero", "ts", "rust", "go", "python"]
}
```

Field rules:

- `slug` must match the directory name.
- `token_budget` is 4x the longest reference implementation rounded up
  to the next 500. Count tokens with `tiktoken` against `gpt-4o`; the
  number does not have to be exact across tokenizers, only generous
  enough that no reference would force more than one attempt for
  reasons of length.
- `wall_time_max_ms` accommodates the slowest language at clean
  install. Rust and Go pay compile costs; budget accordingly.
- When the test cases vary per case (different stdin per case),
  replace `acceptance.stdout_exact` with `"stdout_exact_per_case":
  true` and put expected stdout inside each `tests/*/case-*.json`.
- Use `notes_input_convention` (free-form string field) when input
  routing differs between languages. Zero 0.1.2 has no exposed stdin
  capability for example, so HTTP tasks pass the URL via argv for
  Zero and via stdin for everything else. Document the divergence in
  this field, not in the prompt body.

### prompt.md

The prompt the model receives is built by concatenating
`spec.json#prompt` (the problem statement) with the per-language
scaffold rendered into `{language_scaffold}`. Keep `prompt.md` short:
the title as H1, the prompt text, an Acceptance section, and the
literal `{language_scaffold}` marker on its own line. The scaffold is
generated by `agentlang-spec emit --task <slug> --lang <lang>` from
the companion Zero CLI repo.

### reference implementations

Five files, five languages, **all hand-written**. Model-generated
references would make the benchmark circular: the harness would be
testing whether models can reproduce text that came out of a model.

Each reference must:

1. Compile or interpret with the pinned toolchain in `vendor/<lang>/`.
2. Pass every case in `tests/public/` and `tests/hidden/` byte-exact.
3. Use only the language's standard library. No `tokio`,
   `requests`, `axios`, etc. The HTTP-flavor tasks already stretch
   this; for new tasks, stay in-stdlib.
4. Be production-shape, not contest-golf. Idiomatic Rust uses
   `?`, idiomatic Go uses `errors.Is`, idiomatic Python uses
   context managers. A reference that compiles but reads as
   non-idiomatic gives the grader a bad baseline.

Zero references use the bundled skill data pinned at
`vendor/zero/CURRENT`. When Zero 0.1.x lacks a capability the other
languages have (stdin, generics on user types, some stdlib parsers),
work around it in the Zero reference and capture the workaround in
`notes.md`. Do not pretend the gap does not exist.

### verify.sh

Every task has its own `verify.sh`. It runs each reference, captures
stdout/stderr/exit code, and diffs against the expected output. The
script accepts:

```sh
verify.sh                  # run every language whose toolchain is present
verify.sh --lang zero      # run one language only
```

Languages whose toolchain is missing must print `SKIP: <lang>
(<reason>)` and not fail the script. Failing references must print
`FAIL: <lang> (<reason>)` and contribute to the exit code. Successful
references print `PASS: <lang>`.

The verifier is the day-one acceptance test. A task ships when its
five references all PASS in a clean sandbox with every supported
toolchain present.

### tests/public and tests/hidden

Per-case JSON files following the shape:

```json
{
  "id": "case-NNN",
  "visibility": "public",
  "stdin": "...",
  "expected_stdout": "...",
  "expected_stderr": "",
  "expected_exit_code": 0
}
```

Public cases are included in the prompt the model receives. Hidden
cases are not, and are the basis for the scoring grade. A typical
task has 1-3 public cases and 5-15 hidden cases. Tasks that branch
on input shape (HTTP status families, parser error categories) need
hidden coverage on each branch.

### notes.md

Markdown, 30-100 lines, structured as:

- **Why this exists.** What the task probes about model capability or
  language friction.
- **Calibration.** Per-language gotchas: exact-byte output
  requirements, stdin-versus-argv divergences, why a particular API
  is the right call.
- **Failure modes.** What grader category a typical wrong answer
  lands in (`wrong_output` vs `hallucinated_api` vs `runtime_panic`).

This file is the contributor-facing companion to the spec. It is the
file a maintainer reads when triaging a task scoring oddly across
models.

## running the verifier locally

From a clean clone:

```sh
git clone https://github.com/truffle-dev/agentlang-index
cd agentlang-index
cd corpus/<NNN-slug>
bash verify.sh
```

Expect five PASS lines (one per language) or a mix of PASS and SKIP
when a toolchain is absent. Any FAIL on a clean sandbox is a blocker
for the task.

## running a single-model benchmark

```sh
export OPENAI_API_KEY=sk-...
bun run bench/runner.ts --model gpt-4o-mini
```

`bench/results/<model>/runs/<id>/` accumulates per-task captures.
Aggregate with `bun run bench/aggregate.ts --site` when a sweep
finishes.

The Python harness at `harness/` is the longer-form runner with agent-
loop mode and SQLite storage; see `harness/README.md`.

## PR shape

PRs adding corpus tasks:

- Title: `feat(corpus): add task NNN-slug with five reference impls`
- Body: one paragraph naming the task shape and what it probes, plus
  a `verify.sh` clean-output transcript. No more than ~150 words.
- Branch name: `feat/corpus-NNN-slug`.

PRs improving the harness:

- Title prefix: `feat(harness):`, `fix(harness):`, `fix(bench):`,
  `docs:` as appropriate.
- Body: name the user-visible change and the test that covers it.
  If the change touches the dashboard JSON shape, link the matching
  PR or commit on the dataset side that consumes the new field.

Sign your commits if you can (SSH or GPG, either works). The repo
runs commitlint and a license-header check; both fail loud locally.

## what is out of scope

- **New language slots.** The corpus pins five: zero, ts, rust, go,
  python. Adding a sixth (Java, C++, Swift) is a methodology
  decision, not a routine PR. Open an issue first.
- **New provider adapters past the OpenAI cohort.** Anthropic,
  Google, Together, Fireworks are next in line per the architecture
  doc roadmap. Adapters land in a planned order; please open an
  issue before writing one.
- **Tasks that depend on a third-party library.** The corpus is
  stdlib-only; tasks needing `requests`, `tokio`, `axios`, `gin`,
  `fastapi` would force per-language sandbox provisioning and break
  the operations-cost ceiling.
- **Score-tuning patches.** Changing how the harness counts a
  failure mode is a methodology change. Open an issue, propose the
  change with examples of mis-classified runs from the existing
  dataset.

## one rule that overrides everything

A task that fails any of its own references on day one does not
land. The whole benchmark depends on the references being authoritative.
If a reference cannot pass its own hidden test set in a clean sandbox,
the task is not ready, regardless of how good the spec reads.
